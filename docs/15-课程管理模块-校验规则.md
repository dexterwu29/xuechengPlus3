# 课程管理模块 - 校验规则文档

## 一、概述

本文档描述课程管理模块各接口的请求参数校验规则，包括注解校验（`@Validated` / `@Valid`）与 Service 层业务校验。

**说明**：校验应加在 **DTO**（请求体）上，而非 Entity。Entity 由框架/DB 填充，id、createTime、auditStatus 等由系统自动设置，不做用户输入校验。

---

## 二、课程基础（CourseCreateDTO / CourseUpdateDTO）

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|----------|------|
| name | String | 新增必填 | 非空、长度≤100 | 课程名称 |
| users | String | 否 | 长度≤500 | 适用人群 |
| tags | String | 否 | 长度≤50 | 课程标签 |
| mt | String | 否 | 长度≤20 | 大分类ID |
| st | String | 否 | 长度≤20 | 小分类ID |
| grade | String | 否 | 204001/204002/204003 | 课程等级 |
| teachMode | String | 否 | 200001/200002/200003 | 教育模式 |
| description | String | 否 | 长度≤5000 | 课程介绍 |
| pic | String | 否 | 长度≤500 | 课程图片URL |

**数据字典：**
- grade: 204001 初级、204002 中级、204003 高级
- teachMode: 200001 普通、200002 录播、200003 直播

## 三、课程分页查询（CoursePageQueryDTO）

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|----------|------|
| pageNo | Integer | 否 | 1~9999，默认1 | 页码 |
| pageSize | Integer | 否 | 1~100，默认10 | 每页条数 |
| courseName | String | 否 | 长度≤500 | 课程名称（模糊） |
| auditStatus | String | 否 | 202001/202002/202003/202004 | 审核状态 |
| publishStatus | String | 否 | 203001/203002/203003 | 发布状态 |
| mt | String | 否 | 长度≤20 | 大分类ID |
| st | String | 否 | 长度≤20 | 小分类ID |

**数据字典：**
- auditStatus: 202001未通过 202002未提交 202003已提交 202004通过
- publishStatus: 203001未发布 203002已发布 203003下线

## 四、课程营销（CourseMarketDTO）

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|----------|------|
| charge | String | 否 | 201000/201001 | 收费规则 |
| price | Float | 否 | ≥0 | 现价 |
| originalPrice | Float | 否 | ≥0 | 原价 |
| qq | String | 否 | 长度≤32 | 咨询QQ |
| wechat | String | 否 | 长度≤64 | 微信 |
| phone | String | 否 | 长度≤32 | 电话 |
| validDays | Integer | 否 | 0~9999 | 有效期天数 |

**Service 层业务校验：**
- 当 `charge` 非空且不为 201000/201001 时：抛出业务异常（兜底校验）
- 当 `charge=201001`（收费）时：
  - `price` 必填且 > 0
  - `validDays` 必填且 > 0

## 八、校验失败响应格式

校验失败时返回 HTTP 400，响应体格式：

```json
{
  "code": 400,
  "msg": "name: 课程名称不能为空; grade: 课程等级必须为204001/204002/204003",
  "data": null
}
```

多个字段错误时，以分号分隔。

## 九、实现说明

- **注解校验**：使用 `jakarta.validation`（`@NotBlank`、`@Size`、`@Min`、`@Max`、`@Pattern`、`@DecimalMin` 等）
- **Controller**：在 `@RequestBody` 参数上使用 `@Valid` 触发校验；分页接口 `POST /courses/page` 要求必须传 body（`@RequestBody` 非 required=false）
- **异常处理**：`GlobalExceptionHandler` 捕获 `MethodArgumentNotValidException`、`ConstraintViolationException`，返回统一错误格式
- **Service 层**：复杂业务规则（如收费课程必填价格、有效期）在 Service 中通过 `BusinessException` 抛出；**兜底校验**：`description` 长度≤5000、分页 `auditStatus`/`publishStatus` 枚举值在 Service 中二次校验

---

## 十、与 Entity 校验的区分

| 场景 | 是否校验 | 说明 |
|------|----------|------|
| DTO（用户输入） | ✅ 是 | 在 Controller 层接收时校验 |
| Entity（持久化） | ❌ 否 | id、createTime、auditStatus 等由系统设置，无需用户校验 |
| 枚举值 | 使用项目实际字典 | grade=204001/204002/204003，非 primary/intermediate |
